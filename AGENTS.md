# AGENTS.md

本文档用于给后续 AI、开发者和协作者快速理解“东软熙心健康体检管理系统”的定位、边界、权限和代码规范。修改本项目时必须优先遵守本文档。

## 系统定位

本系统是健康体检管理系统，目标是覆盖需求文档中的核心闭环：

- 用户注册、登录、查看体检套餐、提交体检预约。
- 预约由后端按医生半小时号源自动分配；号源满时进入候补。
- 医生登录后查看预约、确认体检状态、生成体检报告。
- 用户查看自己的预约、候补和体检报告。
- 预约成功、候补递补、报告生成需要给用户发送邮件通知；邮件失败不阻断业务，但必须记录邮件日志。
- 管理员审核医生账号、管理用户和基础配置。

系统不是纯静态演示页。业务数据必须来自后端接口和数据库，不能在页面中写死套餐、预约、报告、用户列表等业务数据。

## 技术栈

- 后端：Go、Gin、GORM、MySQL、Redis、JWT。
- 前端：Vue3、Vite、Element Plus、Vue Router。
- 本地运行：Docker Compose、Makefile。

## 角色与权限

### 用户 user

- 可以注册并登录。
- 可以查看体检套餐。
- 可以创建自己的预约。
- 只能查看自己的预约和报告。
- 不能访问医生端和管理端页面。
- 不能调用医生或管理员接口。

### 医生 doctor

- 可以提交注册，但默认状态为 `pending`。
- 管理员审核为 `active` 后才允许登录。
- 可以查看预约列表。
- 可以更新预约状态。
- 可以生成或更新体检报告。
- 可以查看与预约、报告相关的客户档案。
- 不能审核医生、不能管理系统用户状态。

### 管理员 admin

- 由种子数据初始化，不开放公开注册。
- 可以查看用户列表。
- 可以审核医生账号。
- 可以停用账号。
- 可以访问管理端菜单。

## 认证与安全规则

- 登录接口返回 JWT。
- 前端必须通过 `Authorization: Bearer <token>` 调用受保护接口。
- Redis 保存服务端 session，退出登录时删除 session。
- Redis 同时用于限流等服务端防刷能力。
- 后端必须校验 JWT、Redis session、用户状态和角色权限。
- 前端隐藏菜单只是体验优化，不能作为权限控制依据。
- 用户越权访问他人预约或报告必须返回 `403`。
- 未登录访问受保护接口必须返回 `401`。
- 登录、注册、提交预约、生成报告等操作必须防重复提交。
- 后端启用同 IP 限流，避免恶意或误操作高频请求。
- 密码必须使用 bcrypt 哈希保存，不能明文保存。

## 前端规范

- 登录页、注册页是公开页面。
- 登录成功后根据用户角色进入对应工作台：
  - 用户：`/`
  - 医生：`/doctor`
  - 管理员：`/admin`
- 登录后不能在页面内切换角色或切换端。
- 菜单必须按角色动态生成。
- 父子模块必须清晰归属，禁止把预约创建、我的预约、报告录入、报告查看混在同一个页面里。
- 用户端、医生端、管理员端的页面必须独立表达业务职责，除通用组件外不应复用跨角色页面承载不同语义。
- 所有提交按钮必须有 loading 和禁用态，避免重复点击。
- 搜索输入必须做防抖，提交/状态更新/刷新必须使用提交锁或防抖函数，避免用户连续点击造成重复请求。
- 后端必须同时做限流和事务保护，不能只依赖前端防抖。
- 页面要响应式适配，避免大量固定像素宽度导致窄屏错位。
- 所有业务页面要有表格、筛选、空状态、状态标签和明确操作反馈。
- 不允许放假的流程、假的统计或写死的业务记录。

## 后端规范

- Handler 不应绕过认证中间件读取用户身份。
- 业务接口应从上下文获取当前用户，而不是信任前端传入的 `userId` 或 `doctorId`。
- 用户创建预约时，后端必须使用当前登录用户 ID。
- 医生生成报告时，后端必须使用当前登录医生 ID。
- 管理员审核医生必须校验 `admin` 角色。
- 数据库迁移由 GORM AutoMigrate 完成。
- 种子数据必须可重复执行，不应产生大量重复业务数据。

## 目录说明

### 后端

- `backend/cmd/server`：服务启动入口。
- `backend/internal/auth`：密码哈希、JWT 签发与解析、Redis session key。
- `backend/internal/cache`：Redis 连接。
- `backend/internal/config`：环境变量配置。
- `backend/internal/database`：MySQL 连接和迁移。
- `backend/internal/handlers`：HTTP 路由和业务处理。
- `backend/internal/middleware`：限流等中间件。
- `backend/internal/models`：数据库模型。
- `backend/internal/seed`：模拟数据。

### 前端

- `frontend/src/api`：请求客户端和 token 注入。
- `frontend/src/composables`：共享业务状态、认证状态、防抖工具。
- `frontend/src/components`：通用布局组件和业务组件。
- `frontend/src/layouts`：后台主布局。
- `frontend/src/router`：路由、菜单和角色守卫。
- `frontend/src/views`：登录、注册和各业务页面。

## 前端模块归属

用户端：

- 健康服务：体检套餐、预约体检。
- 我的体检：我的预约、候补状态、我的报告。
- 个人中心：个人资料。

医生端：

- 体检业务：预约处理、报告录入。
- 档案查询：客户档案。

管理员端：

- 用户与权限：用户管理、医生审核。
- 体检服务管理：套餐管理、医生号源管理。
- 系统管理：系统设置、邮件发送记录。

## 预约和候补规则

- 一个医生一个日期下的每个半小时号源容量固定为 1。
- 用户预约时只选择套餐、日期、期望时段和备注，不直接指定医生。
- 后端必须在事务中选择最早可用号源并锁定库存。
- 如果同日期同时段没有可用号源，后端创建候补记录。
- 用户取消已预约且未体检的预约后，后端释放号源并按候补创建时间自动递补。
- 预约成功和候补递补都需要发送邮件通知。

## 邮件配置

- SMTP 配置只能来自 `.env` 或部署环境变量，不能写入源码。
- `.env` 禁止提交；仓库只提交 `.env.example`。
- 邮件发送失败必须记录 `MailLog`，但不能让预约或报告保存失败。

## 交付前检查

每次较大修改后必须至少执行：

```bash
cd backend && go test ./...
cd frontend && npm run build
docker compose up -d --build
make seed
```

还需要验证：

- 未登录访问受保护接口返回 `401`。
- 用户访问医生接口返回 `403`。
- 医生访问预约和报告接口正常。
- 管理员可以审核医生。
- 前端用户、医生、管理员看到的菜单不同。
