# 代码与数据约定

## 后端约定

- Handler 不应绕过认证中间件读取用户身份。
- 业务接口必须从上下文获取当前用户，不信任前端传入的 `userId` 或 `doctorId`。
- 用户创建预约时，后端必须使用当前登录用户 ID。
- 医生生成报告时，后端必须使用当前登录医生 ID。
- 管理员审核医生必须校验 `admin` 角色。
- 数据库迁移由 GORM AutoMigrate 完成。
- 种子数据必须可重复执行，不应产生重复业务数据。
- 业务状态变更必须校验状态流，不能只靠前端禁用按钮。
- 管理员关键操作必须记录操作日志。

## 前端约定

- 登录页、注册页是公开页面。
- 登录成功后按角色进入对应工作台：
  - 用户：`/`
  - 医生：`/doctor`
  - 管理员：`/admin`
- 登录后不能在页面内切换角色或切换端。
- 菜单必须按角色动态生成。
- 父子模块必须清晰归属，禁止把预约创建、我的预约、报告录入、报告查看混在同一个页面里。
- 用户端、医生端、管理员端页面必须独立表达业务职责，除通用组件外不应复用跨角色页面承载不同语义。
- 所有提交按钮必须有 loading 和禁用态，避免重复点击。
- 搜索输入必须做防抖，提交、状态更新、刷新必须使用提交锁或防抖函数。
- 页面要响应式适配，避免大量固定像素宽度导致窄屏错位。
- 所有业务页面要有表格、筛选、空状态、状态标签和明确操作反馈。

## 数据真实性

- 禁止放假的流程、假的统计或写死的业务记录。
- 模拟数据只能放在 seed 中，并且必须能通过接口和数据库真实读取。
- 前端页面展示的数据必须来自后端接口或当前登录态，不得硬编码套餐、预约、报告、用户列表。
