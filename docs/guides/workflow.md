# 开发工作流与提交规则

## 工作流

- 修改前先读 `AGENTS.md` 和本次任务相关 guide。
- 先确认现有实现，再改代码；不要按记忆假设接口、字段或页面仍存在。
- 保持改动范围聚焦，不做无关重构。
- 遇到未提交改动时，默认认为是用户或协作者改动，不得回滚。
- 后端接口语义和前端功能入口不能无意减少。

## 提交规则

- 用户明确要求提交或 push，且验证通过后，可以提交。
- 如果用户没有明确要求提交，但已说明“确认都没问题就可以提交”，在完成验证后可以提交。
- 提交前必须确认 `git status --short`，避免误纳入无关文件。
- 提交信息使用简洁英文或中文，说明真实变更，不写虚假的验证结论。
- push 前确认目标远端。当前 Gitee 远端为 `gitee`。

## 禁止事项

- 禁止提交 `.env`、真实 SMTP 密钥、数据库密码、Redis 密码和任何生产凭据。
- 禁止直接删除源码目录、容器、镜像或远程部署，除非用户明确要求。
- 禁止把测试失败、构建失败或未验证说成已通过。
- 禁止用静态假数据代替后端接口实现。

## 完成态自检

任务收尾前回答：

- 用户要求的功能是否都已完成？
- 相关测试或构建是否已运行，并且结果真实？
- 是否仍有未解决的权限、状态流、数据一致性或假功能风险？
